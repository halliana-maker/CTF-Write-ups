# Lack of Entropy - Firebird Internal CTF 2022 Writeup

**Description:** "Mystiz's computer is lack of entropy. He needs to reuse randomness to generate the primes for RSA..."  

---

## 1. TL;DR
The challenge uses RSA encryption but generates the prime $q$ by taking the base-3 representation of $p$ and re-interpreting it as a base-10 number. Because $p$ and $q$ share exactly the same digits (just in different bases), we can factor $N$ by using a **Depth-First Search (DFS) with Branch and Bound** (or binary search) to guess the digits from the most significant to the least significant, pruning impossible branches instantly.

---

## 2. What Data/File We Have and What is Special
We are provided with a python generator script `chall.py` and an output file `output.txt`. 

**The Script (`chall.py`):**
```python
import random
import gmpy2
# ... (setup) ...
while True:
    p = random.getrandbits(256)
    q = int(gmpy2.digits(p, 3)) # <--- VULNERABILITY

    if not gmpy2.is_prime(p): continue
    if not gmpy2.is_prime(q): continue
    # ... (standard RSA encryption) ...
```

**What is special:**
* **No interactive server:** It's an offline crypto puzzle. We just have $n, e, c$.
* **Flawed Prime Generation:** $p$ is a 256-bit prime. $q$ is generated by converting $p$ to a base-3 string, and directly reading that string as a base-10 integer. For example, if $p$ was $11_{10}$ ($102_{3}$ in base-3), $q$ would be $102_{10}$.

---

## 3. Problem Analysis (In Details)
Let's mathematically define $p$ and $q$. Since $q$ is just the base-3 digits of $p$ treated as base-10, we can express them as polynomials evaluated at different bases:

Let the base-3 digits of $p$ be $d_k, d_{k-1}, \dots, d_1, d_0$, where each $d_i \in \{0, 1, 2\}$.
* $p = \sum_{i=0}^{k} d_i \cdot 3^i$
* $q = \sum_{i=0}^{k} d_i \cdot 10^i$

Since $p$ is a 256-bit integer, its maximum value is $\approx 2^{256} \approx 1.15 \times 10^{77}$. 
How many base-3 digits does $p$ have? 
$k \approx \log_3(2^{256}) \approx 161.5$
So, the highest power $k$ is at most 162.

Our goal is to find the sequence of digits $d_i \in \{0, 1, 2\}$ such that:
$n = \left(\sum_{i=0}^k d_i 3^i\right) \times \left(\sum_{i=0}^k d_i 10^i\right)$

---

## 4. Initial Guesses / First Try
Because we are solving a polynomial where the coefficients are unknown but strictly bounded ($d_i \in \{0, 1, 2\}$), one might initially think of Coppersmith's method or Lattice reduction (LLL). However, treating 162 distinct digits as variables in a lattice is mathematically complex and highly likely to fail. 

A better approach is a **Branch and Bound Search** (often conceptually linked to Binary Search, as the flag cleverly hints). Since we know the maximum length of the number, we can guess the digits one by one starting from the Most Significant Digit (MSD). 

---

## 5. Exploitation Walkthrough / Flag Recovery

If we guess a digit $d_k$, we can calculate the **strict lower and upper bounds** for the final product $n$. 
For the remaining unknown digits (from $k-1$ down to $0$), their minimum possible value is when they are all `0`, and their maximum is when they are all `2`.

**Bounds calculation at step $k$:**
* `Max remaining for p` = $\sum_{i=0}^{k-1} 2 \cdot 3^i = 3^k - 1$
* `Max remaining for q` = $\sum_{i=0}^{k-1} 2 \cdot 10^i = \frac{2}{9}(10^k - 1)$

If the actual target $n$ does not fall between our `current_p * current_q` (lower bound) and `(current_p + max_p) * (current_q + max_q)` (upper bound), we immediately discard that digit choice. This trims the search tree massively, allowing us to find $p$ and $q$ in milliseconds.

    def dfs(k, P_val, Q_val):
        # Base case: all digits processed
        if k < 0:
            if P_val * Q_val == n:
                return P_val, Q_val
            return None
            
        # Try each possible base-3 digit
        for d in[0, 1, 2]:
            P_new = P_val + d * (3**k)
            Q_new = Q_val + d * (10**k)
            
            # Calculate bounds for remaining digits
            P_lo = P_new
            P_hi = P_new + 3**k - 1
            
            Q_lo = Q_new
            Q_hi = Q_new + (10**k - 1) * 2 // 9
            
            # Prune branches (Bounds check)
            if P_lo * Q_lo <= n <= P_hi * Q_hi:
                res = dfs(k - 1, P_new, Q_new)
                if res:
                    return res
        return None

    print("[*] Starting DFS branch and bound to recover p and q...")
    
    # 256-bit max digits in base-3 is ~162. 
    # Starting at k=165 guarantees we cover it safely.
    result = dfs(165, 0, 0)
    
    if result:
        p, q = result
        print(f"[+] Found p = {p}")
        print(f"[+] Found q = {q}")
        
        # Standard RSA Decryption
        phi = (p - 1) * (q - 1)
        d = pow(e, -1, phi)
        m = pow(c, d, n)
        
        flag = long_to_bytes(int(m)).decode('utf-8')
        print("[+] Flag:", flag)

if __name__ == "__main__":
    solve()

## 6. What We Learned
* **Don't Reuse Entropy:** Cryptographic algorithms assume complete independence and unpredictability of underlying variables. By directly mapping $p$ to $q$, the entropy of the system dropped catastrophically.
* **Search Algorithms in Crypto:** While CTF crypto often focuses on heavy algebra (Lattices/Elliptic Curves), fundamental algorithmic techniques like Branch & Bound or Binary Search can break weakly generated keys much faster than black-box solvers like Z3. The strict algebraic bounds generated by the bases ($3^k, 10^k$) made pruning wildly effective.
