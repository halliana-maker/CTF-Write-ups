## 3. Problem Analysis (In-Depth)

### Vulnerability 1: Overdetermined LWE (The "e" Oracle)
The LWE implementation has $N=77$, but gives us up to $100$ queries. Since $100 > 77$, the system is strictly overdetermined. Because the error $e \in [0, 3]$ is incredibly small compared to the 77-bit modulus $q$, we can easily recover the exact error vector using Kannan's Embedding technique and LLL lattice reduction. Once $e$ is stripped, finding $s$ is just solving a linear system of equations over $GF(q)$.

### Vulnerability 2: Coordinate Truncation & Overlap Bypass
The LWE private key `s` is concatenated into a massive integer `priv` ($\approx 77 \times 77 = 5929$ bits). 
When we send a point, the server does:
```python
point_int &= (1 << 512) - 1  # 512-bit truncation
x, y = unpack_point(point_int) # two 256-bit integers
x ^= priv
y ^= priv
```
Because `x` and `y` are only 256 bits, `x ^ priv` effectively only alters the bottom 256 bits via XOR, while the upper 5600+ bits of `priv` are simply added!
Mathematically, evaluated modulo the curve's $p$:
$X_{eval} \equiv (priv_{high} + (x \oplus priv_{low})) \pmod p$

To inject a specific target coordinate $X_{target}$, we isolate $x$:
$x = ((X_{target} - priv_{high}) \pmod p) \oplus priv_{low}$

### Vulnerability 3: The Invalid Curve Attack
Looking closely at the custom ECC implementation in `util.py`:
```python
m = (3 * x1 * x1 + self.a) * pow(2 * y1, -1, self.p) % self.p
```
The slope calculation `m` relies *only* on the parameter `a`. It completely ignores the parameter `c` in the standard Weierstrass equation $y^2 = x^3 + ax + c$. Furthermore, the server never verifies if the coordinates we provide actually lie on the original curve. 
This means if we send a point $(X, Y)$ that belongs to a different curve $y^2 = x^3 + ax + c'$, the server will happily perform valid scalar multiplication on that *new* curve.

## 4. Initial Guesses & First Tries (The Pitfalls)

During exploitation, a "naive" theoretical script will crash and burn. Here is what happened in our initial iterations:

1. **The 50-Minute LLL Hang:**
   Our first attempt fed the raw $177 \times 100$ basis matrix (99 queries + 77 dimensions + 1 vector) directly into `matrix.LLL()`. Because the rows were linearly dependent, SageMath suffered from extreme coefficient explosion, taking almost an hour to reduce the lattice.
   *Fix:* We computed the Row Echelon form over $GF(q)$ first to extract the exact $100 \times 100$ independent image basis. This dropped the LLL execution time from **50 minutes to 0.05 seconds**.
2. **The BSGS "Infinite" Factorization Freeze:**
   We generated random invalid curves and grabbed any small prime factor $f$ from the curve order. However, if the order contained $f^k$ (where $k > 1$), Sage's `discrete_log` would silently panic, attempt to fully factor a 255-bit integer natively, and hang forever.
   *Fix:* We strictly filtered our precomputed curves ensuring the factor's multiplicity was exactly $1$ (`e_pow == 1`), mathematically isolating the prime-order cyclic subgroup.
3. **Network Bottlenecks:**
   Doing `io.send` and `io.recv` 99 times per round took over 3 minutes due to basic TCP round-trip latency. 
   *Fix:* **Pipelining.** We concatenated all 99 inputs into a single byte-string `b"m\n0\n" * 99` and blasted it in one packet, processing the outputs in a fast loop.

## 5. Exploitation Walkthrough / Flag Recovery

Here is the exact step-by-step path the final script took to execute the exploit in under 15 seconds:

### Step 1: Precomputing Weak Curves
Before touching the server, the script offline-generated 17 "invalid" elliptic curves sharing the server's `a` parameter, but with random `c` parameters. It mapped curves whose orders contained distinct prime factors $m_i$ between $10,000$ and $2,000,000$. We accumulated these until $\prod m_i > 2^{240}$ (covering the 30-byte bounds of the secret).

### Step 2: The LWE CVP Lattice Reduction
For Round $i$:
1. We sent the pipelined 99 queries of `m = 0`.
2. Extracted the vectors $A$ and scalars $b$. Because $m=0$, we knew the message payload was just the deterministic hash of "0".
3. We built the $100 \times 100$ reduced Kannan's embedding matrix and ran LLL.
4. The shortest vector returned the exact LWE error bounds $e$. We subtracted $e$ from $b$ to get perfect linear equations, solved them over $GF(q)$, and rebuilt the massive `priv` integer.

### Step 3: Subverting the Coordinate Truncation
With `priv` known, we selected our precomputed weak-curve point $(X_i, Y_i)$ of order $m_i$. We applied our overlap bypass math:
```python
priv_low = priv & MASK
P_H_mod = (priv - priv_low) % p
x_send = (X_i - P_H_mod) % p ^ priv_low
```
We packed `x_send` and `y_send` and fired it at the exchange oracle.

### Step 4: Discrete Log & CRT
The server returned the multiplied point. We mapped it back to our precomputed invalid curve $E_i$ and used Baby-Step Giant-Step (`discrete_log()`) to find the scalar $k_i$.
This gave us: 
`master_sec` $\equiv k_i \pmod{m_i}$

### Step 5: Flag Decryption
After iterating this 17 times, we fed all our $k_i$ and $m_i$ arrays into the Chinese Remainder Theorem (`crt(rems, mods)`). This cleanly output the exact 30-byte `master_sec`!
We hashed the master secret to recreate the AES key, decrypted the ECB cipher, and obtained the flag:

## 6. What We Learned
- **Network Pipelining is King:** A theoretically sound script is useless if latency makes it take hours. Blasting inputs in a single TCP frame is mandatory for heavy I/O crypto challenges.
- **Algebra before LLL:** Never blindly throw rectangular, overdetermined matrices into `.LLL()`. Always filter linearly dependent rows by mapping the row space over the finite field first to save extreme computational overhead.
- **Strict Subgroup Containment:** When abusing $y^2 = x^3 + ax + c$, knowing the order has a small prime isn't enough for efficient BSGS. You must ensure the subgroup is strictly cyclic (multiplicity of 1) or your computational frameworks will freeze attempting to traverse ambiguous generator structures.
