# Write-up: SpecialBase (rev)

**Challenge Author:** shiguang  
**Category:** Reverse Engineering   
**Flag Format:** `PUCTF26{[a-zA-Z0-9_]+_[a-fA-F0-9]{32}}`   
**Description:** *Base is basic for ctfers. But do you really know how it works?*   

---

## 1. TL;DR
The binary is a stripped Linux ELF that takes our input, encodes it using a **custom Base64 algorithm**, and compares it against a hardcoded string. Instead of using a statically defined custom alphabet, the binary uses a **custom Virtual Machine (VM)** to dynamically generate the 64-character alphabet at runtime from a 94-character charset. By reverse-engineering the VM's bytecode, we can simulate the alphabet generation in Python and reverse the custom Base64 encoding to recover the flag.

---

## 2. Problem Analysis (Details)
When we open the binary in Ghidra and inspect the strings, we notice a few interesting things:
1. A 94-character string:   
```ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/!@#$%^&*()-_~`[]{}|;:',.<>?```
2. A target ciphertext string:  
```SX#P#(]*mV^h>&#Mr2XJ`|X@<#0qdhd*]@X6mM?zm2]Z<&d:]2S.m|S.]MK.m&]t<M?zdVZ=```

Looking at the main checking logic (`FUN_0010395e`), the program reads up to 255 characters of input, passes it to an encoding function, and compares the result with the target ciphertext. 

Diving deeper into the initializers and the encoding setup (`FUN_0010336c`), the program does not simply map bytes to standard Base64. Instead, it prepares an array of bytecode instructions and passes them to a massive `switch-case` block (`FUN_0010136a`). This function represents a **custom VM dispatcher**, complete with virtual registers, memory addressing, arithmetic operations, and conditional jumps. 

The VM's execution is divided into two distinct stages:
*   **Stage 1:** It iterates over the 94-character base string and scrambles it into a new 94-character array using modulo arithmetic and linear probing to handle collisions.
*   **Stage 2:** It pseudo-randomly selects 64 characters from the scrambled string to serve as the final translation table for the Base64 encoding. The randomness is provided by a Linear Congruential Generator (LCG) using standard glibc constants: `next = (current * 0x41c64e6d + 0x3039) & 0x7fffffff`.

Finally, the 64-character alphabet is used in standard Base64 6-bit grouping to encode our input.

---

## 3. Initial Guesses / First Try
At first glance, seeing a 94-character string immediately brings `Base85`, `Ascii85`, or `Base94` to mind. I initially suspected the binary was taking our input and converting it via a standard Base85 variant.

However, after comparing the length of the expected output (72 chars, excluding the `=` padding) and looking at the loop structures inside the encoding function, the `>> 6` and `& 0x3F` bitwise operations were a dead giveaway. This was firmly a **Base64** encoding scheme, which requires a 64-character alphabet. The 94-character string was merely the "pool" from which the final alphabet was dynamically drafted. The complex array manipulations and switch-cases indicated that simple string-replacement brute-forcing wouldn't work—I had to completely reverse the dynamic generation.

---

## 4. Exploitation Walkthrough / Flag Recovery

To solve the challenge, we need to extract the exact 64-character alphabet generated by the VM. Rather than trying to dump it dynamically via GDB (which could be annoying due to anti-debugging or memory complexities), we can statically emulate the VM's logic in Python.

**Step 1: Reconstruct the PRNG**
The LCG is clearly visible in `FUN_00101344`. We can easily write this in Python:
```python
def next_rand(v):
    return (v * 0x41c64e6d + 0x3039) & 0x7fffffff
```

**Step 2: Simulate Stage 1 (The Scramble)**
By analyzing the VM's bytecode logic, the algorithm loops 94 times. It calculates an index `(i * 0x61 + length) % length` and inserts the character. If the slot is taken, it uses a linear scan (starting from 0) to find the next empty slot.

**Step 3: Simulate Stage 2 (The Selection)**
Next, the VM selects 64 characters. It relies on the LCG to pick indices. If an index is already chosen, it calls `next_rand()` again and steps through the array using modulo additions until it finds an unselected character.

**Step 4: Decode!**
Once we have the generated alphabet, recovering the flag is as simple as reversing the bitwise Base64 operations. 

Here is the complete Python script to generate the alphabet and decode the flag:

```python
def next_rand(v):
    return (v * 0x41c64e6d + 0x3039) & 0x7fffffff

def get_alphabet():
    charset = "ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz0123456789+/!@#$%^&*()-_~`[]{}|;:',.<>?"
    length = len(charset)
    
    # Stage 1: Scramble the 94-char charset
    out1 = [0] * length
    aux1 = [0] * length
    for i in range(length):
        r6 = (i * 0x61 + length) % length
        if aux1[r6] != 0:
            curr = 0 
            while aux1[curr] != 0:
                curr = (curr + 1) % length
            r6 = curr
        out1[r6] = ord(charset[i])
        aux1[r6] = 1
    scrambled = "".join(chr(x) for x in out1)
    
    # Stage 2: Pick 64 chars using the LCG
    final_table = [0] * 64
    aux2 = [0] * length
    r5 = 0 # Initial PRNG seed
    for j in range(64):
        r5 = next_rand(r5)
        r7 = (r5 % length + j) % length
        if aux2[r7] != 0:
            r5 = next_rand(r5)
            r11 = 0
            r12 = r5 % length
            while True:
                r11 = (r11 + r12) % length
                if aux2[r11] == 0:
                    break
            r7 = r11
        final_table[j] = scrambled[r7]
        aux2[r7] = 1
        
    return "".join(final_table)

def decode_custom_b64(target, alphabet):
    if target.endswith('='):
        target = target.split('=')[0]
    val = 0
    bits = 0
    decoded =[]
    
    char_to_idx = {char: i for i, char in enumerate(alphabet)}
    
    for char in target:
        val = (val << 6) | char_to_idx[char]
        bits += 6
        while bits >= 8:
            decoded.append((val >> (bits - 8)) & 0xFF)
            bits -= 8
            
    return bytes(decoded).decode()

if __name__ == "__main__":
    target = "SX#P#(]*mV^h>&#Mr2XJ`|X@<#0qdhd*]@X6mM?zm2]Z<&d:]2S.m|S.]MK.m&]t<M?zdVZ="
    alphabet = get_alphabet()
    print(f"[*] Generated Alphabet: {alphabet}")
    
    flag = decode_custom_b64(target, alphabet)
    print(f"[+] Recovered Flag: {flag}")
```

## 5. What We Learned
* **Custom VM Architectures:** Recognizing the typical `switch-case` inside an infinite loop as a fetch-decode-execute loop is crucial for modern RE. 
* **Dynamic Asset Generation:** Reversing algorithms that build their encryption constants dynamically at runtime rather than storing them locally inside `.rodata`. This effectively defeats generic tools like CyberChef or static strings analysis.
* **PRNG Patterns:** Identifying common constants (like `0x41c64e6d` and `0x3039`) quickly reveals the use of a standard Linear Congruential Generator.
