
# Write up: Empty Hook
**Category:** Pwn  
**Author:** shiguang  
**Description** You know there is an empty hook waiting for you? Hint: The path to flag is /flag   
**Protection:** Full RELRO, PIE Enabled, NX Enabled, No Stack Canary  

## 1. TL;DR
The challenge involves bypassing a strict Seccomp sandbox and a custom shellcode filter. The exploit chain is:
1. **Info Leak:** Exploit an uninitialized stack variable/out-of-bounds `write` to leak a randomly generated byte and a stack address.
2. **Control Flow Hijacking:** Exploit a buffer overflow using a **1-byte partial PIE overwrite** to redirect execution to a hidden function that reads our payload into `.bss`.
3. **Hook Satisfaction:** Forge the required magic values in the `.bss` segment to pass the program's custom "hook" checks.
4. **Shellcode Execution:** Upload an XOR-encoded **ORW (Open, Read, Write)** shellcode. Since the binary statically checks for and blocks the `syscall` opcode (`0x0f 0x05`), we bypass this by calling `syscall@plt` to retrieve the flag.

---

## 2. Problem Analysis (Details)
When analyzing the binary statically (via Ghidra) and dynamically, we observe several distinct phases:

1. **The Setup & Canary:** 
   The program generates a random byte via `syscall(0x13e)` (getrandom) and stores it on the stack alongside a stack pointer (`rbp-0x30`). It then attempts to zero out the buffer, but leaves this crucial data intact.
2. **The Leak (`write` OOB):**
   The program asks for input and uses `read(0, buf, 0xff)`. Afterwards, it does a `write(1, buf, 0x108)`. Because it writes out more bytes than it reads, it bleeds the leftover stack data containing the random byte and the stack pointer.
3. **The Vulnerability (`FUN_001015d0`):**
   The program calls a function that reads `0x200` bytes into a `128-byte` buffer. This is a classic Buffer Overflow. However, PIE is enabled, and we don't have a code base leak yet.
4. **The "Hook" & Shellcode Decoder (`FUN_00101600`):**
   Before exiting, the program checks global variables in the `.bss` segment (`0x40e0` and `0x40e8`). If `0x40e0` equals the magic number `0xb136804f`, it acts as a shellcode loader. It allocates executable memory, XOR-decodes our payload using the random byte leaked earlier, and ensures no `0x0f 0x05` (syscall) instructions are present.
5. **The Jail (Seccomp):**
   Finally, the program applies a strict Seccomp filter via `prctl` that prevents `execve`, forcing us to use an ORW approach to read `/flag`.

---

## 3. Initial Guesses / First Try
* **Format String?** The binary parses our input looking for `%` and specifically checks for `%n`, rejecting it. A format string attack is a dead end.
* **Standard ret2libc?** We have a buffer overflow, but we don't have a libc leak, and Seccomp drops `execve` (so `system("/bin/sh")` won't work anyway).
* **The "Empty Hook" hint:** The name implies we need to fill an empty hook structure. Looking at the `.bss` checks, the binary *wants* to execute shellcode for us, but the `.bss` section is completely empty. We need a way to write to `.bss` before the program reaches the hook execution phase.

## 4. Exploitation Walkthrough / Flag Recovery

### Step 1: The Information Leak
To start, we need to bypass the decoder's XOR key and prevent a crash during the overflow. We send `0xff` bytes of padding. The `write(1, buf, 0x108)` outputs our padding plus the leaked stack data.

```python
r.sendafter(b"say:\n", b"A" * 0xff)
leak = r.recvn(0x108)
local_38 = u64(leak[0x100:0x108])

uVar10 = local_38 >> 56 # The random XOR key
stack_addr = local_38 & 0xffffffffffffff # Stack address
saved_rbp = stack_addr + 0x130 # Calculating original RBP
```

### Step 2: The 1-Byte Partial Overwrite
We have a buffer overflow of `0x200` into a `0x80` buffer. Because PIE is enabled and we don't know the base address, we can't overwrite the whole return address. 
Instead, we do a **1-byte partial overwrite**. The original return address points to `main+offset` (e.g., ending in `...12e1`). We overwrite the Least Significant Byte (LSB) with `\xe3`. This perfectly redirects execution to a skipped block of code (`call 1590`) which reads `0x900` bytes directly into the `.bss` section!

We also carefully overwrite the saved RBP with our leaked `saved_rbp` to ensure the program doesn't segfault when it eventually returns to `main`.

```python
pad = b"A" * 0x80
pad += p64(saved_rbp)
pad += b"\xe3" # 1-byte PIE overwrite
r.sendafter(b"QAQ\n", pad)
```

### Step 3: Forging the Hook
Now the program is reading `0x900` bytes into `.bss`. We must satisfy the hook constraints:
* `[bss+0x80] == 0xb136804f` (Magic check)
* `[bss+0x88] == sc_size` (Size of our shellcode)

We also must manually XOR-encode our shellcode using the random byte (`uVar10`), matching the exact offset and step algorithm the binary uses to decode it.

### Step 4: Opcode Evasion & ORW Shellcode
We need to `open`, `read`, and `write` the flag. However, the decoder scans our un-XOR'd shellcode and will terminate the program if it finds `0x0f 0x05` (the standard x64 `syscall` instruction).
To bypass this, we use the fact that the binary's PLT contains `syscall@plt`. We calculate the PIE base using `call/pop`, calculate the address of `syscall@plt`, and call it using the standard C-calling convention (`rdi=sysno, rsi=arg1, rdx=arg2`).

```assembly
call get_rip
get_rip:
    pop rbp
    sub rbp, 0x1749             /* rbp = PIE Base */
    lea r15,[rbp + 0x1130]     /* r15 = syscall@plt */

    /* openat(AT_FDCWD, "/flag", 0, 0) */
    mov edi, 257                /* sysno: openat */
    push -100; pop rsi          /* arg1: AT_FDCWD (-100) */
    lea rdx, [rip + flag_str]   /* arg2: path */
    xor ecx, ecx                /* arg3: flags = O_RDONLY */
    xor r8d, r8d                /* arg4: mode */
    call r15

    /* ... proceed to read() and write() using 'call r15' ... */
```

We send the encoded payload, the program decodes it, passes the security checks, executes our PLT-based shellcode, and prints the flag!

## 5. What We Learned
This challenge was a fantastic compilation of intermediate-to-advanced binary exploitation techniques:
1. **1-Byte Partial PIE Overwrites:** When you lack a code base leak, overwriting just the last byte of a return address is a powerful way to jump to nearby, unexecuted code branches (like error handlers or hidden inputs).
2. **Information Bleeding:** Over-reading/Over-writing buffers (`write` using `0x108` on an array of smaller size) is a classic way to leak stack canaries, pointers, and crypto keys.
3. **Opcode Filtering Bypass:** Static analysis sandboxes that ban specific assembly bytes (like `0x0f 0x05`) can be easily defeated if the binary was compiled dynamically with those instructions already present in its `.plt` or `.text` segments. You simply pivot to the existing `syscall@plt` wrapper.
