# ghostcrypt - BSidesSF CTF 2026 Write-Up 

## 1. TL;DR
**Vulnerability:** Weak Keyed Substitution Cipher + Cryptogram Pattern Exfiltration
**Solution:** Interacting with the server's API reveals that the cryptography used is a simple 1-to-1 substitution cipher. By fetching the secret "Litany" ciphertext and applying classic pattern matching (crib dragging) to the word structures, we can decrypt the message without needing the secret "Arcana" key.

---

## 2. What Data/Files We Have & What is Special

We are provided with a web URL and its underlying HTML/JS source code. 

**The Frontend:**
The web application is heavily themed as a spooky, interactive "summoning" board. It features:
*   Horror-themed ambient background sounds and flashing lightning effects.
*   An image stack that flashes individual letters of the alphabet on the screen using a Polybius-square-style or grid-based image set (`/grid/a.avif`, etc.).
*   Two main input fields: **Incantation** (Plaintext) and **Arcana** (Key).
*   A **"Summon Litany"** button that explicitly states the flag format: `FLAG: 'CTF{<summon litany>}'. Interpret litany as all lowercase, no spaces.`

**The API Interaction (Client/Server):**
Looking at the JavaScript, the frontend communicates with the backend via two main endpoints:
1.  `POST /invoke`: Takes `incantation` and `arcana` from the form and returns a `playback_sequence` array of characters.
2.  `POST /summon-litany`: Requires no parameters. Returns a secret `playback_sequence` generated by the server with a hidden arcana (`?????????`).

*What's special here is the distraction:* The challenge dresses up a classic cryptography problem in a highly polished, distracting audio-visual interface. The core of the challenge entirely lives in the JSON arrays returned by the backend.

## 3. Problem Analysis (In Details)

By completely ignoring the frontend flashing lights and focusing on the API, we realize this is a chosen-plaintext attack (CPA) scenario. We have an encryption oracle (`/invoke`) and a target ciphertext (`/summon-litany`).

To understand the cipher, we can write a quick Python script to probe the `/invoke` endpoint:

```python
import requests
BASE_URL = "https://ghostcrypt-2c7aa570.challenges.bsidessf.net"

# Probe with empty arcana
r1 = requests.post(f"{BASE_URL}/invoke", data={"incantation": "abcdefgh", "arcana": ""})
print("Enc (key=''):", "".join(r1.json()['playback_sequence']))

# Probe with an arcana key
r2 = requests.post(f"{BASE_URL}/invoke", data={"incantation": "abcdefgh", "arcana": "ghost"})
print("Enc (key='ghost'):", "".join(r2.json()['playback_sequence']))
```

**Server Output:**
```text
[*] Plaintext:  abcdefgh
[*] Enc (key=''): ABCDEFGH
[*] Enc (key='ghost'): MOLNPBDA
```

**Conclusion:** 
1. The length of the plaintext matches the length of the ciphertext (1-to-1 mapping). 
2. Changing the "Arcana" shuffles the alphabet entirely.
3. This is a standard **Keyed Monoalphabetic Substitution Cipher**.

## 4. Initial Guesses / First Try

Our first thought might be to figure out *how* the Arcana generates the alphabet (e.g., does "ghost" place G-H-O-S-T at the start of the alphabet and shift the rest?). However, to get the flag, we need to decode the "Litany".

Let's pull the Litany from the server:
```python
r = requests.post(f"{BASE_URL}/summon-litany")
print("Litany:", "".join(r.json()['playback_sequence']))
```
**Output:** `HONHONL RNKH PQD CDZKFE`

Since we don't know the server's secret Arcana (it displays as `?????????`), we could try to brute-force short Arcana keys against the `/invoke` endpoint. But wait—the ciphertext contains distinct word lengths and repeating character patterns. Because it's a simple substitution cipher, we can just solve it like a newspaper cryptogram!

## 5. Exploitation Walkthrough / Flag Recovery

We have the ciphertext: **`HONHONL RNKH PQD CDZKFE`**

Let's break it down using classic cryptanalysis and pattern matching:

**Step 1: Analyze the first word `HONHONL`**
*   Length: 7 letters.
*   Pattern: `1-2-3-1-2-3-4` (The first three letters repeat).
*   Common English words with this exact structure: `MURMURS`, `SINGING`, `TARTARS`.
*   Let's assume it's **`MURMURS`**.
    *   `H` = M
    *   `O` = U
    *   `N` = R
    *   `L` = S

**Step 2: Analyze the second word `RNKH`**
*   Length: 4 letters.
*   Applying our knowns: `R` is mapped to ?, `N` = R, `K` = ?, `H` = M.
*   Structure: `_ R _ M`
*   A very common 4-letter English word fitting this is **`FROM`**.
    *   `R` = F
    *   `K` = O

**Step 3: Analyze the third word `PQD`**
*   Length: 3 letters.
*   It has no known letters yet, but `PQD` in a phrase like "MURMURS FROM ___" is almost guaranteed to be **`THE`**.
    *   `P` = T
    *   `Q` = H
    *   `D` = E

**Step 4: Analyze the fourth word `CDZKFE`**
*   Length: 6 letters.
*   Applying our knowns: `C`=?, `D`=E, `Z`=?, `K`=O, `F`=?, `E`=? (Wait, we know `D` maps to `E`, let's not mix ciphertext/plaintext).
*   Let's map what we know from `FROM` and `THE`:
    *   `K` = O
    *   `D` = E
*   Cipher text: `C D Z K F E` -> Plain text: `_ E _ O _ _`
*   Looking for a spooky 6-letter word ending in something fitting the horror theme: **`BEYOND`**.
    *   `C` = B
    *   `D` = E (Matches our previous assumption perfectly!)
    *   `Z` = Y
    *   `K` = O (Matches perfectly!)
    *   `F` = N
    *   `E` = D

**Verification:**
The phrase is **"murmurs from the beyond"**. All letter mappings are mathematically consistent with a 1-to-1 substitution cipher. No letters map to conflicting plaintext characters.

**Formatting the Flag:**
The instructions on the frontend explicitly stated: *FLAG: 'CTF{<summon litany>}'. Interpret litany as all lowercase, no spaces.*

Applying the formatting rules:
`murmurs from the beyond` -> `murmursfromthebeyond`

**Flag:**
`CTF{murmursfromthebeyond}`

## 6. What We Learned
1. **Don't get distracted by the UI:** The flashy CSS animations, audio context APIs, and scary sounds were essentially a smokescreen. The actual attack surface was a simple JSON API.
2. **Always check the constraints of the cipher first:** Before writing a brute-force script to recover the key algorithm, do a basic analysis of the ciphertext. If a substitution ciphertext retains spaces and word lengths, human pattern recognition (crib dragging) will often solve it in seconds.
3. **Chosen Plaintext Oracles are great for fingerprinting:** Sending `abcdefgh` to the server instantly proved it was a substitution cipher and not a Polybius/Bifid variant, saving a massive amount of time.
